# Git Commit Convention

## 规则

每次 git commit 前必须遵守以下规范：

### 0. 何时 commit（必须）

**开发过程中禁止提交 commit。** 完成开发后，按以下流程处理：

1. 列出本次开发中所有修改过的文件
2. 询问用户：「以上文件是否需要提交 commit？」
3. 等待用户确认后再执行提交

**禁止**在开发中途自行提交，即使某个阶段已"完成"。

### 0.1 文件范围（必须）

**只提交本次需求直接涉及的文件。** 对每个变更文件，判断其修改是否完全属于本次需求：

- **仅本次需求修改** → 可以提交
- **被多个需求共同修改**（即文件中混有其他需求的改动）→ **禁止提交该文件**，并告知用户：「文件 `<path>` 包含多个需求的改动，请自行拆分后手动提交」

**禁止**将多个需求的改动混入同一 commit，即使改动在同一文件内。

### 1. 前缀（必须）

| 前缀 | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `refactor` | 重构（不改变功能） |
| `chore` | 构建、依赖、工具等杂务 |
| `docs` | 文档 |
| `test` | 测试 |
| `perf` | 性能优化 |
| `style` | 代码风格（不影响逻辑） |
| `ci` | CI/CD 配置 |

### 2. 提交信息格式（必须）

**第一行必须包含 issue 编号**，格式为 `<前缀>(<ISSUE>): <一句话总结>`。

正文逐条说明本次变更要点（中文，一行一条）。每行格式为 `<类型><序号>: <说明>`：

```
<前缀>(<ISSUE>): <一句话总结>

<类型>1: <变更要点>
<类型>2: <变更要点>
```

- `<ISSUE>`：Jira/工单编号，如 `BCS-448`（不含 `#`）
- **第一行 `<前缀>`**：本次 commit 的主类型（通常与主要改动一致，见前缀表）
- **正文 `<类型>`**：与该条变更的实际性质一致，使用与第一行相同的前缀关键词（`feat`、`fix`、`refactor` 等），**不限于与第一行相同**
  - 同一类型多条：序号递增，如 `feat1`、`feat2`
  - 不同类型可混写：例如第一行为 `feat(...)` 时，正文可同时写 `feat1:` 与 `fix1:`（feature 中夹带 bug 修复时）
- `feat1`、`feat2` 仅为示例写法，**不是**要求正文只能写 feat 类内容
- 每条应写清**做了什么**；若与背景强相关，可在同条内简要说明**为什么**
- 仅一条时写对应类型即可，如 `feat1:` 或 `fix1:`

**示例（纯 feature）：**

```
feat(BCS-448): ask-user 工具重构为三通道

feat1: 将原单通道 ask-user 拆分为 CLI、HTTP、SDK 三通道入口
feat2: 统一三通道的请求/响应模型，便于后续扩展
```

**示例（feature 中夹带 fix）：**

```
feat(BCS-448): ask-user 工具重构为三通道

feat1: 将原单通道 ask-user 拆分为 CLI、HTTP、SDK 三通道入口
fix1: 修复 HTTP 通道在空 body 时返回 500 的问题
```

### 3. 扫描关联文档（必须）

提交前必须检查 `.ai/` 和 `docs/` 目录，将与本次变更相关的文档一并纳入同一 commit：

- **`.ai/plans/`**：实施计划文档
- **`.ai/specs/`**：设计/规格文档
- **`docs/`**：用户文档、说明文档

判断方式：文档文件名或内容与本次改动的功能/模块相关，即视为关联。

**禁止行为**：不得将代码与其配套文档拆成两次 commit（除非文档在代码提交后才新增）。

### 4. Issue 号（必须）

- **每条 commit 必须在第一行标题中写明 issue 号**，格式：`<前缀>(<ISSUE>): ...`
- **禁止**将 issue 号单独写在正文末尾（如 `关联 #1024`）
- 如果不确定 issue 号：**立即停下，询问用户**，不得假设或省略
- 询问格式：「这次提交需要关联 issue 号，请提供（如 BCS-448）」

## 禁止行为

- 不得在开发过程中擅自提交 commit，必须完成后询问用户
- 不得提交包含多个需求改动的文件，需提醒用户自行处理
- 不得使用英文写 commit message（前缀关键词除外）
- 不得省略正文条目（至少一条，格式为 `<类型><序号>:`）
- 不得在不知道 issue 号的情况下提交（必须先问用户）
- 不得使用 `--no-verify` 绕过检查
- 不得遗漏 `.ai/`、`docs/` 下的关联文档

## 红色警报 — 立即停下的念头

| 这个念头 | 实际含义 |
|----------|----------|
| "我帮用户顺手提交一下" | 禁止。必须等用户确认。 |
| "issue 号不重要，先提了再说" | 禁止。必须先问用户。 |
| "这个文件顺便也加进来" | 先判断是否属于本次需求。 |
| "文档下次再提交" | 禁止拆分。关联文档必须同批提交。 |
